# VLA 推理能跑多快？用 VLA-Perf 解密推理性能全景

> 原标题：How Fast Can I Run My VLA? Demystifying VLA Inference Performance with VLA-Perf
> 作者：Wenqi Jiang, Jason Clemons, Karu Sankaralingam, Christos Kozyrakis
> 发表：arXiv:2602.18397v1, 2026
> 链接：https://arxiv.org/abs/2602.18397
> 代码：https://github.com/NVlabs/vla-perf

---

## 一、研究背景与动机

- VLA（Vision-Language-Action）模型在具身 AI 任务中展现出强大能力，但**部署到真实机器人时面临严格的实时推理约束**——典型 RGB 相机帧率为 24-60 Hz，模型推理必须跟上这一节奏。
- 当前社区在推理效率方面已有诸多探索（小模型、量化、层跳跃、减少去噪步数、异步推理、双系统架构等），但**缺乏对 VLA 推理性能全景的系统性理解**。根本原因在于模型架构与推理系统的组合空间巨大：模型端涉及规模、架构、上下文长度、动作块大小等；系统端涉及硬件（边缘 GPU vs 数据中心 GPU）、推理位置（本地 vs 边缘服务器 vs 云端）、网络条件（有线 vs WiFi vs 蜂窝网络）。
- 本文的目标是回答一个根本性问题：**如何设计未来的 VLA 模型和推理系统以支持实时推理？** 为此需要一种能快速、低成本地分析任意模型-系统组合的性能预测工具。

## 二、核心贡献

1. **VLA-Perf 分析工具**：提出首个基于 Roofline 模型的 VLA 推理性能分析框架，可预测任意模型-系统组合的最优推理延迟和吞吐量，无需实际部署即可快速评估。
2. **首次系统性性能研究**：利用 VLA-Perf 对模型设计（规模扩展、架构选择、长上下文、异步推理、双系统管线）和系统部署（硬件能力、推理位置、网络性能）进行了全面评估。
3. **15 条关键结论（Takeaways）**：从海量评估中提炼出 15 条可操作的设计指导，覆盖模型设计和系统部署两大维度。

## 三、方法原理

### 3.1 整体框架

VLA-Perf 将 VLA 推理抽象为**模型组件序列 + 数据传输**的流水线。一个典型 VLA 系统包含三个模型组件：

- **视觉编码器（Vision Encoder）**：将相机图像编码为视觉 token
- **VLM 骨干网络（VLM Backbone）**：整合视觉 token 和语言指令，生成 KV 缓存
- **动作专家（Action Expert）**：基于 KV 缓存生成动作预测

根据各组件的部署位置（本地或服务器），组件之间通过本地传输或网络传输交换数据（原始图像、视觉 token、KV 缓存、动作预测）。

**输入参数**分为两类：
- **模型参数**：视觉编码器/VLM/动作专家的选择、序列长度、去噪步数、动作块大小、动作维度等
- **系统参数**：加速器峰值算力和内存带宽、网络上下行带宽和延迟

### 3.2 关键技术细节

**延迟计算模型**：端到端推理延迟建模为所有模型组件推理延迟与所有数据传输延迟之和：

$$T_{total} = \sum_{m \in M} T_m + \sum_{d \in D} T_d$$

其中 $M$ 为模型组件集合，$D$ 为数据传输阶段集合。

每个模型组件 $m$ 的延迟为其所有算子延迟之和：

$$T_m = \sum_{o \in O_m} T_o$$

每个算子 $o$ 的执行延迟使用 **Roofline 模型**估算，取计算延迟和内存访问延迟的较大值：

$$T_o = \max\left(\frac{FLOPs_o}{FLOP/s_h},\ \frac{Bytes_o}{MemBW_h}\right)$$

直觉解释：算子的执行时间由**计算密度**决定——如果计算量大于内存搬运时间，则为计算瓶颈（Compute-bound）；反之为内存瓶颈（Memory-bound）。

**网络传输延迟**建模为：

$$T_d^{net} = NetLat + \frac{Bytes_d}{NetBW}$$

即基础延迟加上数据量除以带宽。

**关键抽象**：VLA-Perf 假设每个模型组件在单个加速器上执行（现代 GPU 已具备足够显存），但不同组件可以部署在不同加速器上。

### 3.3 模型保真度验证

通过与 Ma et al. 在 RTX 4090 上用 Triton 优化实现的 π0 实测数据对比，VLA-Perf 的预测达到了 **73.3%~82.6%** 的保真度（即实测延迟为 Roofline 上界的 73%~83%），且负载越大保真度越高。差距来源于硬件微架构细节和软件开销（kernel 启动延迟、OS 干扰等），VLA-Perf 估算的是性能上界。

## 四、实验与结果

### 4.1 实验设置

- **基准模型**：π0 架构（400M SigLIP 视觉编码器 + 2B Gemma VLM + 300M 扩散动作专家），3 个相机，14 DoF 双臂操作，输入序列 800 token，默认动作块大小 50、去噪步数 10
- **扩展模型**：π0-L (9.1B)、π0-XL (16.7B)、π0-XXL (81.3B)，通过替换更大的 SigLIP/Llama2/更大动作专家构建
- **硬件**：Jetson Thor（边缘）、RTX 4090（消费级）、A100/H100/B100（数据中心）
- **网络**：Ethernet 1G/10G、WiFi 6/7、4G/5G、慢/快云端
- **指标**：端到端推理延迟（ms）和吞吐量（Hz）

### 4.2 主要结果

**模型设计维度的关键发现：**

| 结论编号 | 核心内容 |
|---------|---------|
| **T1** | 数据中心 GPU 已可为小型 VLA（如 π0）实现相机帧率级推理（A100: 61.7 Hz, B100: 314.4 Hz），但边缘 GPU 仍受限（Jetson Thor: 19 Hz） |
| **T2** | 动作预测在所有硬件上均为内存瓶颈（OI=54.0），视觉和 VLM 在除 Jetson Thor 外的 GPU 上为计算瓶颈 |
| **T3** | 各组件延迟随模型规模**近似线性增长** |
| **T4** | B100 在 81B 模型上仍可达 9.6 Hz（实时），RTX 4090 在 16.7B 时已爆显存 |
| **T5** | 长上下文 VLA：B100 支持 1K 历史帧（11.7 Hz），但 10K 帧降至 1.2 Hz；边缘/消费级 GPU 仅支持约 100 帧 |
| **T6** | **去噪步数对延迟影响极大**（10→50 步使整体延迟增 2.15×），**动作块大小影响极小**（50→250 仅增 11%），因为动作预测是内存瓶颈 |
| **T7** | 带动作块的扩散 VLA 比经典自回归 VLA 快 1-2 个数量级（π0: 3.2 ms vs OpenVLA 式: 327.6 ms） |
| **T8** | 自回归仅在生成少量 token 或启用并行解码时有竞争力 |

**系统部署维度的关键发现：**

| 结论编号 | 核心内容 |
|---------|---------|
| **T9** | 服务器端推理（即使消费级 RTX 4090 + WiFi）几乎总优于本地推理，除非网络极差（4G 或更低） |
| **T10** | 设备-服务器协作推理通常比纯本地推理还慢（KV 缓存下载开销大），且总是比纯服务器推理慢，实用价值有限 |
| **T11** | 异步推理在慢网络下显著提升吞吐（WiFi 7: 2.63×, 5G: 5.99×, 慢云端: 13.79×），但不降低延迟 |
| **T12** | 双系统（System 1 + System 2）的增益取决于硬件和网络：B100 + Ethernet 10G 下 2.24× 提升，5G 下仅 1.05× |
| **T13** | 本地推理：Jetson Thor 已可达 10 Hz，100 Hz 需模型级优化（缩小模型/减少去噪步数/量化） |
| **T14** | 边缘服务器：10 Hz 用消费级 GPU + 无线网络即可；100 Hz 需数据中心 GPU + 快网络 |
| **T15** | 云端推理：10 Hz 需好网络；100 Hz 必须启用异步推理 |

### 4.3 关键性能数据

**π0 在不同硬件上的推理延迟：**

| 硬件 | 视觉延迟 | VLM 延迟 | 动作延迟 | 总延迟 | 频率 |
|------|---------|---------|---------|-------|------|
| Jetson Thor | 6.06 ms | 20.30 ms | 26.20 ms | 52.57 ms | 19.0 Hz |
| RTX 4090 | 4.02 ms | 19.79 ms | 7.25 ms | 31.06 ms | 32.2 Hz |
| H100 | 0.71 ms | 3.30 ms | 2.14 ms | 6.15 ms | 162.5 Hz |
| B100 | 0.40 ms | 1.87 ms | 0.91 ms | 3.18 ms | 314.4 Hz |

**瓶颈分析**：视觉编码器和 VLM 的算子强度（OI）分别为 321.4 和 542.8 FLOPs/Byte，在大多数 GPU 上为计算瓶颈；动作专家 OI 仅 54.0，始终为内存瓶颈。Jetson Thor 因使用 LPDDR 内存（270 GB/s vs B100 的 8000 GB/s），所有组件均为内存瓶颈。

## 五、局限性与展望

- **作者讨论的局限**：
  - 仅聚焦于操作任务的 VLA 模型，未覆盖自动驾驶、四足机器人、无人机等其他具身 AI 领域
  - 未纳入机器人执行延迟和传感器延迟（因平台差异大），不是端到端系统性能分析
  - 分析基于 Roofline 上界，实际性能约为预测的 73-83%，且假设了最优软件实现
- **潜在改进方向**：
  - 扩展至更多具身 AI 领域（自动驾驶、四足等），这些场景有不同的系统约束（如更强调本地推理）
  - 整合推理、传感、执行的端到端系统性能分析
  - 针对特定硬件和软件平台校准 VLA-Perf，提升预测精度

## 六、灵魂三问

1. **它解决了什么问题？** 之前各团队在不同硬件、不同网络、不同模型上各自测推理速度，结果零散且不可比较，无法回答"应该用多大的模型、部署在哪里、需要什么网络"这类系统设计问题；现在 VLA-Perf 提供了一个统一的分析框架，可以在几秒内预测任意模型-系统组合的推理性能，并从中提炼出 15 条可操作的设计准则。

2. **为什么这么做？** 核心选择是用**解析式 Roofline 模型**而非逐一实测。不实测的原因很直接——模型规模×架构×硬件×网络的组合空间是爆炸性的，穷举实验在成本和时间上都不可行。Roofline 模型虽然只估算性能上界（实际约为 73-83%），但在当前对 VLA 推理性能理解极度匮乏的阶段，粗粒度的定量估算已足以提供有价值的设计指导。

3. **什么证据最有说服力？** Takeaway 6 的去噪步数 vs 动作块大小对比：去噪步数从 10 增到 50 导致总延迟增 2.15×，而动作块大小从 50 增到 250 仅增 11%。这直接揭示了扩散动作专家的内存瓶颈本质——增加 token 数几乎不增加延迟（因为瓶颈在加载模型权重和 KV 缓存而非计算），具有非常强的设计指导意义：可以放心增大动作块以平滑执行，而减少去噪步数才是降低延迟的有效杠杆。

## 七、个人总结

1. **最核心的 idea**：VLA-Perf 的价值不在于某个具体的性能优化技术，而在于**将 VLA 推理性能从零散的经验判断变为系统性的定量分析**。通过 Roofline 模型统一抽象模型和系统参数，它使得"什么模型配什么硬件配什么网络能达到多少 Hz"这类问题可以快速回答。

2. **最大的优势和不足**：优势在于覆盖面极广——从 2.7B 到 81B 模型、从边缘到云端、从有线到 4G/5G，15 条 Takeaways 几乎覆盖了 VLA 部署的所有关键决策点，实用价值很高。不足在于分析完全基于推理延迟，未涉及推理延迟与任务精度之间的权衡（如减少去噪步数确实降低延迟，但精度损失多少？）。此外 Roofline 模型假设最优实现，实际工程中的优化差距可能因平台而异。

3. **对后续研究的启发**：本文提供了一张 VLA 推理性能的"地图"，后续工作可以在此基础上定向优化。例如，动作专家始终为内存瓶颈这一发现暗示了内存带宽优化（如量化、稀疏化）比计算优化更关键；设备-服务器协作推理效果差的结论也提醒社区，分布式推理方案需要重新审视 KV 缓存传输的开销。
